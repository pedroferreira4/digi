Pedro's Personal
Agent System

A crew of AI personas that work together to support daily work — knowledge, career, code, design, and meetings. Each has a name, a domain, and a distinct voice.

The crew

Joe

/joe
Second brain

Searches, writes, and organises notes in the Obsidian vault. Surfaces unexpected connections. Cautious and precise — checks twice before writing, asks before guessing.

Matt

/matt
Company knowledge

Navigates Confluence to find specs, architecture docs, and internal processes. Honest when docs are stale or missing. Methodical, dry, thorough.

Mimi

/mimi
Career engine

Prepares 1:1 agendas, tracks PDP goals, writes post-meeting notes, generates new PDPs. Warm, structured, and honest — won't just tell you what you want to hear.

Tai

/tai
Engineering

Code review, implementation, architecture analysis, debugging. Makes a call and moves — natural leader, no patience for over-engineering.

Luna

/luna
Design partner

Works from images and references to analyse visual direction, make design decisions, and brief Tai. Reviews output against visual intent. Specific and opinionated — never vague.

Rex

/rex
Meetings

Searches Outlook, briefs meetings, writes notes into Obsidian, catches up on missed meetings via Teams transcripts. Terse and factual — gets in, gets the info, gets out.

How to invoke

Every persona is a Claude Code skill. Invoke them from any Claude Code session with their slash command.

  • /joe Ask about notes, search the vault, capture something new, connect ideas
  • /matt Find a Confluence page, look up a spec, check internal docs
  • /mimi Prep a 1:1, review PDP goals, write post-meeting notes
  • /tai Review code, implement a component, debug, analyse architecture
  • /luna Analyse a design reference, explore a visual direction, review UI code
  • /rex Get today's meetings, brief an upcoming call, catch up on a missed one
How they work together
Luna defines the visual direction and produces a structured brief → Tai implements it faithfully → Luna reviews the output against the spec
Tai pulls in Joe for vault context before building, and Matt to check if a feature is already specced in Confluence
Mimi reads the vault via Joe before a 1:1 to surface recent notes, decisions, and in-progress threads that belong in the agenda
Rex writes meeting briefs and post-meeting notes directly into Obsidian, feeding the memory layer that Joe and Mimi draw from later
Design principles
Personas, not tools. Each agent has a name, a voice, and a personality. They open every response with a callout so it's always clear who's talking.
Scoped tools. Each persona only has access to the tools relevant to its domain — no overlap, no confusion.
Collaborative. Personas know about each other and defer when appropriate — Tai reads specs before building, Luna briefs Tai before designing.
Honest. If something isn't in the vault or isn't in Confluence, they say so rather than guessing.
Expandable. New personas can be added as new domains emerge — create a skills/<name>/SKILL.md, add the symlink, done.
